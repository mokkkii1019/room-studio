## 解決済み: 自動切り抜きによる待ち時間と誤切り抜き

2026-07-20 に `AUTO_CUT=false`（`room-studio.html`）で自動経路を停止し、本番へデプロイ済み。
**約2分の待ちと「人物だけが切り抜かれる」は本番から消えている。**

塞いだのは `cutQueueAdd`（収集時のキュー、`queuePendingCuts` 経由も含む）と、`placeFromLib` の
配置時の優先切り抜き・ブラウザ内AIフォールバック。**手動の「AIで背景を切り抜く」(`fAiCut`) と
保存済み `cutCanvas` の読み込みは従来どおり動く。**

止めた理由（どちらも実測）:

| 問題 | 実測値 |
|---|---|
| `/bgcut` 1枚（キャッシュMISS） | **約7秒**（6.9〜7.6s を4回計測。HITは0.37秒） |
| 50件収集（クライアント同時実行3） | **約2分** |
| 家具カテゴリの候補画像に人物が含まれる割合 | 約17%（212枚のラベル付けでは8%） |

切り抜き精度が破綻していた理由: ISNet は **salient object detection（最も目立つ物体を
切り出す）** モデル。人物がソファに座っている販促画像では**人物が最も目立つ物体**なので、
モデルは正しく動作した結果として**人物だけを残し家具を捨てる**（「KUFUL SOFA」で検証済み）。
閾値調整では直らない。モデルの用途が違う。

**画像選別が効くまでこの既定を戻さないこと。** 悪い画像を切り抜いても価値はない。

### 残っている設計上の無駄（未着手）
`/bgcut` は 1024×1024 で推論して **300〜600KB の RGBA PNG** を返しているが、クライアントは
**alpha チャンネルしか使っていない**（`room-studio.html` の `applyCutAlpha`）。
マスクだけを小さく返せば推論・エンコード・転送すべてが軽くなる。手動切り抜きの7秒に効く。

## 実施済み（デプロイ済み・main にマージ済み）

コミット `5770dff` / マージ `29bb722`。**これらは待ち時間には効いていない**。家電カテゴリの
「意図しない画像」には劇的に効く。

### A. 候補画像URLの当て推量を廃止
`_numbered_variants()`（画像URLの連番を推測して候補を水増し）を削除。

| | 実測 |
|---|---|
| 推量URLの生存率 | **9/105 = 8.6%**（91%が404） |
| 実在URL（`mediumImageUrls`）の生存率 | **60/60 = 100%** |
| 候補数 | 8.2枚/件 → **2.6枚/件** |

クライアントは表示前に全候補をロードするため、404の応答待ちが収集前半の待ちの主因だった。

### B. アクセサリ混入を楽天APIの `NGKeyword` で除外
「液晶TV 保護フィルム」が `TYPE_MATCH` の「液晶テレビ」にマッチして通過していた。

| カテゴリ | 変更前 | 変更後 |
|---|---|---|
| tv | 30件中 **25件が保護フィルム（83%）** | 0件 |
| washing_machine | 8件が交換用フィルター | 0件 |
| vacuum | 3件が収納スタンド | 0件 |

`NGKeyword` はスペース区切り＝**OR-of-exclusions**（いずれかを含めば除外）。実機で検証済み。
ローカル除外だけだと tv が30件中3件まで枯渇するため、取得元で弾くのが必須。

**注意（検証で捕まえた誤り）**:
- 「フィルム」は **tv 限定**にすること。`mirror` の「フィルムミラー（割れない鏡）」は正当な商品。
- `carpet` の「マット」も正当（ラグマット）。除外してはいけない。
- `vacuum` の「スタンド」も単体で弾かないこと。「スタンド付き」は本体。

## 実測データ（再取得のコストが高いので記録）

### 候補画像の中身（212枚を目視ラベル付け）

ラベル付き評価セットは `tools/collect-audit/eval-sets/{dev,holdout}.json` に**URLとラベルだけ**
committed（楽天規約により画像は保存しない。`eval_imgscore.py` が実行時に再取得する）。

- dev: sofa / chair / coffee_table / shelf / bed — 40アイテム120候補
- holdout: desk / chest / dining_table / curtain — 32アイテム92候補（**閾値調整には使わない**）

| 候補の内訳 | 割合 |
|---|---|
| テキストバナー・レビュー・仕様書 | 62% |
| バリエーション表・コラージュ | 17% |
| 人物入り | 8% |
| 生地のアップのみ（curtain） | 4% |
| **使える商品写真（`clean`+`ok`）** | **18%** |

### 🔴 いちばん重要な数字: そもそも使える候補が無いアイテムが6割

| | dev(40件) | holdout(32件) |
|---|---|---|
| 3候補のうち**1枚でも使える**アイテム | **40%** | **34%** |

楽天の商品検索APIが返す画像は `mediumImageUrls` の最大3枚で、**これ以上増やせない**
（連番推測は91%が404だったので廃止済み）。つまり**どんなに選別を賢くしても上限は約4割**。
ここを超えるには「候補が全部ダメなアイテムは捨てて、そのぶん多めに取る」しかない。

### 候補スコアラーの性能（アイテム単位＝3候補から選んだ1枚が使える画像か）

| | dev | holdout |
|---|---|---|
| 旧 `scoreSingleItem` | 18% | 19% |
| 新 `scoreCandidate` | **35%** | **31%** |
| 上限 | 40% | 34% |

新スコアラーは上限の88%(dev)/91%(holdout)を回収している。**選別ロジック側の伸びしろは
ほぼ無い。** 残る差は上限そのもの。

### 旧スコアラーが壊れていた理由（同じ罠を踏まないこと）
- 「縁が無地で明るい＋中央にコントラスト」を加点していたが、これは**カラーバリエーション表**
  （白背景・整然・中央に商品が並ぶ）に最高得点を与える。実測でも採用したい画像の中央値
  -0.055 に対し捨てたい画像 +0.158 と**符号が逆**だった。
- 肌色ペナルティは人物ではなく**木目・ベージュ生地**に反応していた（人物 0.128 < 採用したい
  画像 0.177）。撤去済み。
- → **白背景を褒めてはいけない。** 新スコアラーは減点のみで構成している。

### ❌ 「判定は128pxでも精度が落ちない」— これは誤りだった（記録として残す）
dev/holdout のアイテム単位の的中率で測ると 600px:35% / 128px:32% とほぼ差が出なかったため、
一度128pxで本番投入した。**この測定が不適切だった。** 正しくは「使える画像とテキストの
スコア差」が128pxでは -0.02（＝判別できていない）まで潰れている。正しい実測と根拠は
下の「判定解像度は300px」節を見ること。

**教訓: アイテム単位の的中率は1件の増減が8〜9ポイント動くので、この種の劣化を検出できない。**
解像度・閾値を触ったら `delivered` セットの precision@K で測ること。

### マスク統計による判定の可否（2026-07-19 の調査。参考）
| 画像種別 | coverage | blob数 | 検出可否 |
|---|---|---|---|
| バリエーション表 | 0.301 | **12** | ✅ 検出できた |
| カラーチップ格子 | **0.069** | 1 | ✅ 検出できた |
| 白背景の単体ソファ | 0.215 | 1 | 正常と判定 |
| **テキストバナー** | — | — | ❌ **素通りする** |

→ マスク統計は切り抜きを走らせないと得られない上、テキストを見逃す。現行スコアラーは
切り抜き不要でテキストを直接数えるので、この経路はもう使っていない。

## 実装済み: 多めに取ってスコア上位だけ採用する（over-fetch → top-K）

`COLLECT_OVERFETCH=3`（`room-studio.html`）。要求数の3倍（上限90＝`/collect` の上限）を取得し、
**全件を300pxに落として採点 → スコア上位 count 件だけ原寸で読む**。

固定閾値での足切りも実装前に測ったが、**在庫が枯れて要求数に届かないだけで精度は同じ**
だったので採用しなかった:

| 方式 | dev(40件) | holdout(32件) |
|---|---|---|
| 全件採用（＝選別なし） | 100%納品 / 35% | 100%納品 / 31% |
| **3倍取得→上位1/3** | **100%納品 / 62%** | **100%納品 / 64%** |
| 3倍取得→閾値 -1.2 | 77%納品 / 70% | 100%納品 / 64% |
| 4倍取得→上位1/4 | 100%納品 / 70% | 100%納品 / 75% |

→ **閾値は使わない。上位K件を採るだけでよい**（枯れないぶん素直に優れている）。

### 🔴 判定解像度は300px。下げてはいけない（2026-07-20 に一度失敗している）
当初これを128pxにした。理由は「サムネイルでも精度は落ちない」という測定結果だったが、
**その測定が間違っていた**。アイテム単位の的中率という粗い指標で測ったため、テキスト検出が
壊れていることを検出できなかった。本番で配信された48枚を採点し直して判明した実測:

| 判定に使う画像 | 1枚 | 上位24枚の精度 | 「使える画像 − テキスト」の平均差 |
|---|---|---|---|
| 128px | 4.8KB | 46% | **-0.02 ← テキストの方が高得点。判別できていない** |
| 200px | 9.6KB | 54% | +0.38 |
| **300px** | **18.8KB** | **62%** | **+0.46** |
| 600px（原寸） | 58.1KB | 62% | +0.41 |

縮小するとグリフの微細構造が潰れ、文字が「小さく密な連結成分」として検出できなくなる。
実例: 多くの店舗が使う「THANK YOU! 累計販売台数◯万台突破」バナーは、128pxだと48枚中
5〜22位に居座るが、300pxなら39〜45位に落ちる。

**教訓: 解像度・閾値を変えたら `delivered` セットで測ること。** アイテム単位の指標は
1件の増減が8〜9ポイント動くので、この種の劣化を検出できない。

### ⚠ 枚数スライダーが大きいと効果が薄れる
`/collect` の上限が90件なので、倍率は要求数に依存する:

| 枚数スライダー | 取得数 | 倍率 | 期待精度 |
|---|---|---|---|
| 15 | 45 | 3.0倍 | 約62〜64% |
| 30 | 90 | 3.0倍 | 約62〜64% |
| **50（既定）** | 90 | 1.8倍 | 約45〜56% |
| 80 | 90 | 1.1倍 | ほぼ選別なし |

**既定値50を30以下に下げるかは運営判断（未確認）。** 上限90は楽天3ページ＝サーバレスの
実行時間制約から来ており、`_provider_official.search()` の `page <= 3` が根拠。

- **次の注力先 → 画像選別**。スコアラー刷新 + over-fetch/top-K まで完了。

## 実際に配信されている画像の実測（`delivered` セット）

本番から240件取得してスコア上位48枚を目視ラベル付けした結果。**判断境界そのもの**を測るので、
これが最も実態に近い:

| 失敗の内訳（128px時代の上位48枚中24枚が失敗） | 枚数 |
|---|---|
| テキストバナー | 14（うち同一の「THANK YOU」テンプレが6） |
| コラージュ・バリエーション表 | 6 |
| 接写・素材アップ | 4 |

300pxに直した後、新規データで独立に再測定 → 上位24枚中14枚が使える（**58%**）。
失敗10枚の内訳は **接写・素材アップ5、テキスト5**。

### 🔴 いまの最大の失敗要因は「接写・素材アップ」
天板の角、扉の側面、生地のアップなど。**商品の実写だが切り抜き素材にならない。**
スコアの平均は `closeup` = -0.33 で、`ok`（-1.02）や `text`（-1.46）より**高い＝最上位に来る**。

減点方式なので「何も減点対象が無い画像」が満点になるのが構造的な原因。以下は試して
**効かなかった**ので、同じ手を繰り返さないこと（`delivered` セットで測定済み）:

- 大きな文字を多段解像度で検出する項 → 上位24枚の精度 15→15。接写には無効
- 「長い輪郭が無い画像を減点」(`long_frac`) → 接写は天板の長い輪郭を持つので効かない
- 「中央が周辺より情報量が多くない画像を減点」(`subject`) → 誤差範囲、良い画像も巻き添え

必要なのは「商品が**全体として**写っているか」の判定で、これは現在の特徴量では届いていない。

## メーカー/ショップ指定について（2026-07-20 調査）

運営から「フリーワードに `art of black` を入れてソファを収集したが、その店の商品が
ほとんど来ない」との指摘。調査結果:

- **収集自体は正しかった。** 返った11件は**全件 ART OF BLACK**だった。
- 少なく見えた理由は、**楽天上に条件に合うソファが11件しか無い**こと（ショップコード指定でも
  同じ11件）。参考: 同店は `chair` なら62件ある。
- 「その店の商品に見えなかった」理由は候補画像の選択ミスで、これは上の128px問題と同一原因。

### ただし別の欠陥がある: ショップコードが使われていない
`/collect` は `shop=`（楽天 shopCode）を受け取れるのに、**クライアントは一度も送っていない**。
メーカー選択UIですら、ショップ名を**キーワードとして**渡している（`room-studio.html` の
`mode==='brands'` の分岐）。キーワード一致に依存するため、商品名に店名が入っていない
商品は取りこぼす。`State.makers` は shopCode を持っているので、渡すだけで直せる。
今回の art of black では件数は変わらなかったが、店によっては効くはず。**未着手。**

## 次にやること

1. **接写・素材アップの判定**（上の🔴）。いまの最大要因。
2. **ショップコードを `/collect` に渡す**（上記）。メーカー選択が名前一致頼みになっている。

## 計測スクリプト（`tools/collect-audit/`）

Vercelにはデプロイされない（`.vercelignore` で `tools/` を除外済み）。

| スクリプト | 用途 |
|---|---|
| `eval_imgscore.py <cache> [--parity]` | **候補スコアラーの評価（dev/holdout/delivered）。閾値・解像度を触ったら必ず実行** |
| `eval-sets/delivered.json` | **配信された上位48枚のラベル。判断境界を測る最重要セット** |
| `imgscore.py` | スコアラーのPython版（`room-studio.html` と同一ロジック） |
| `parity.js` | `room-studio.html` から関数を抜き出して Python と数値一致を検証 |
| `fetch_samples.py <out> <types> <per>` | カテゴリ別に候補画像を実取得してサンプル作成 |
| `sheet.py <dir> <per>` | 番号付き contact sheet を生成（目視ラベル付け用） |
| `audit_titles.py <types> <out.json>` | タイトルだけ取得してアクセサリ混入を監査 |
| `eval_exclude.py <titles.json>` | 除外語の効果と**誤検出**を出力（出荷前に必ず実行） |
| `test_ng_matrix.py` | 楽天 `NGKeyword` の意味論と語の効きを実機検証 |
| `verify_search.py` | パッチ後の `search()` をカテゴリ横断で検証 |
| `verify_prod.py` | **デプロイ後の本番** `/collect` を検証 |

Vercelにはデプロイされない（`.vercelignore` で `tools/` を除外済み）。

`eval_imgscore.py` は `.env` もAPIキーも不要（画像CDNから直接取得する）。要 `numpy` /
`scipy` / `Pillow`、`--parity` は `node`。キャッシュ先は任意のテンポラリでよく、実行後に
消してよい:

```
python tools/collect-audit/eval_imgscore.py /tmp/imgcache --parity
```

`fetch_samples.py` など楽天APIを叩くものは `.env` の `RAKUTEN_APP_ID` /
`RAKUTEN_ACCESS_KEY` と `RAKUTEN_REFERER=https://roomstudio.jp/` が必要。楽天APIは
約1QPS制限があるので、連続実行時は `time.sleep(3)` を挟むこと（429 を実際に踏んだ）。

**サンプル画像は一切保存していない**（楽天規約：画像を保存しない）。`eval-sets/*.json` に
URLとラベルだけを持ち、評価のたびに再取得する。画像が消えていた場合はその1枚を
スキップして件数を報告する。
