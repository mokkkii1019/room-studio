## いま本番で起きている問題（未解決・最優先）

### 1. 切り抜き精度が破綻している
ISNet は **salient object detection（最も目立つ物体を切り出す）** モデル。人物がソファに
座っている販促画像では**人物が最も目立つ物体**なので、モデルは正しく動作した結果として
**人物だけを残し家具を捨てる**。実例を検証済み（「KUFUL SOFA」の画像でソファが完全に消失）。

- 閾値調整では直らない。モデルの用途が違う。
- 家具カテゴリの候補画像の**約17%に人物**が含まれる。
- 販促バナーやバリエーション表を切り抜いた結果は**元画像より悪い**。
- それを収集した全アイテムに自動適用してサムネイルを差し替えている。

### 2. 待ち時間が約2分（改善されていない）
| 項目 | 実測値 |
|---|---|
| `/bgcut` 1枚（キャッシュMISS） | **約7秒**（6.9〜7.6s を4回計測） |
| 50件収集（クライアント同時実行3） | **約2分** |
| キャッシュHIT | 0.37秒 |

2026-07-19 の変更（下記）は**切り抜きキューに一切触れていない**ため、この待ちは残ったまま。
むしろ候補選定が変わって選ばれる画像が変わり、温まっていたエッジキャッシュが無効化されて
MISSが増えた可能性が高い。

### 3. 設計上の無駄
`/bgcut` は 1024×1024 で推論して **300〜600KB の RGBA PNG** を返しているが、クライアントは
**alpha チャンネルしか使っていない**（`room-studio.html` の `applyCutAlpha`）。
マスクだけを小さく返せば推論・エンコード・転送すべてが軽くなる。7秒の内訳として無視できない。

## 実施済み（デプロイ済み・main にマージ済み）

コミット `5770dff` / マージ `29bb722`。**これらは待ち時間には効いていない**。家電カテゴリの
「意図しない画像」には劇的に効く。

### A. 候補画像URLの当て推量を廃止
`_numbered_variants()`（画像URLの連番を推測して候補を水増し）を削除。

| | 実測 |
|---|---|
| 推量URLの生存率 | **9/105 = 8.6%**（91%が404） |
| 実在URL（`mediumImageUrls`）の生存率 | **60/60 = 100%** |
| 候補数 | 8.2枚/件 → **2.6枚/件** |

クライアントは表示前に全候補をロードするため、404の応答待ちが収集前半の待ちの主因だった。

### B. アクセサリ混入を楽天APIの `NGKeyword` で除外
「液晶TV 保護フィルム」が `TYPE_MATCH` の「液晶テレビ」にマッチして通過していた。

| カテゴリ | 変更前 | 変更後 |
|---|---|---|
| tv | 30件中 **25件が保護フィルム（83%）** | 0件 |
| washing_machine | 8件が交換用フィルター | 0件 |
| vacuum | 3件が収納スタンド | 0件 |

`NGKeyword` はスペース区切り＝**OR-of-exclusions**（いずれかを含めば除外）。実機で検証済み。
ローカル除外だけだと tv が30件中3件まで枯渇するため、取得元で弾くのが必須。

**注意（検証で捕まえた誤り）**:
- 「フィルム」は **tv 限定**にすること。`mirror` の「フィルムミラー（割れない鏡）」は正当な商品。
- `carpet` の「マット」も正当（ラグマット）。除外してはいけない。
- `vacuum` の「スタンド」も単体で弾かないこと。「スタンド付き」は本体。

## 実測データ（再取得のコストが高いので記録）

### 候補画像の中身（96枚を目視ラベル付け）
`mediumImageUrls` の3枚は**大半が販促物**。クリーンな単体商品写真は:

| カテゴリ | クリーンな商品写真の割合 |
|---|---|
| sofa | 約25%（24枚中6枚） |
| chair | **約10%**（24枚中2〜4枚） |
| tv | ほぼ全滅（当時は保護フィルム混入のため。Bで改善済み） |

内訳: テキストバナー（「楽天ランキング1位」「安心の1年保証」等）、バリエーション/カラー表、
人物入り販促画像、テキスト重ねの商品写真。

### マスク統計による判定の可否
切り抜きマスクの形状から画像の良し悪しをある程度判定できる:

| 画像種別 | coverage | blob数 | 検出可否 |
|---|---|---|---|
| バリエーション表 | 0.301 | **12** | ✅ 検出できた |
| カラーチップ格子 | **0.069** | 1 | ✅ 検出できた |
| 白背景の単体ソファ | 0.215 | 1 | 正常と判定 |
| **テキストバナー** | — | — | ❌ **素通りする** |

→ マスク統計だけでは不十分。テキスト量の検出と人物検出が別途必要。

## 未決の判断（運営に確認中だった。ここから再開）

運営がPCを落とす前に回答が得られなかった2点。

### 判断1: 現状の自動切り抜きをすぐ止めるか
- **(推奨) 自動切り抜きをOFFにする** — 収集時の自動キューを停止し元画像を使う。待ち時間
  約2分が即座に消え、人物だけの切り抜きも出なくなる。手動の「AIで背景を切り抜く」ボタンは
  残すので、ユーザが選んだときだけ実行される。失われるのは「収集した全アイテムが勝手に
  透過になる」挙動だけ。
- 判定を実装してから自動継続する（実装完了まで悪い挙動が本番に残る）
- 現状維持で設計を固めてからまとめて変更

### 判断2: 次の注力先
- **(推奨) 画像選別を極める** — 人物・バナー・バリエーション表を除き、白背景の単体商品写真
  だけを採用する。**クリーンな画像なら切り抜き自体がほぼ不要**になるため、品質と速度を
  同時に解決する。「悪い画像を速く切り抜いても価値はない」。
- 事前計算で待ち時間を消す（切り抜き精度が悪いままだと悪い結果を先回りで作ることになる）
- 切り抜き自体を見直す（ISNetをやめる／マスクのみ返す軽量化）

## 再開時の進め方（推奨）

1. 上の判断1・2を運営に確認する。
2. 判断1が「OFF」なら先に実施（`room-studio.html` の `cutQueueAdd` / `queuePendingCuts` を
   無効化。手動ボタンの経路 `aiCutCanvas` は残す）。出血を止めるのが先。
3. 画像選別の判定器を作る。**必ず先にラベル付きサンプルで評価してから本番に入れること**
   （今回、未検証の除外語で `mirror` を壊しかけた実例がある）。
4. 選別が効いてから、事前計算・マスクのみ返す軽量化に進む。

## 計測スクリプト（`tools/collect-audit/`）

Vercelにはデプロイされない（`.vercelignore` で `tools/` を除外済み）。

| スクリプト | 用途 |
|---|---|
| `fetch_samples.py <out> <types> <per>` | カテゴリ別に候補画像を実取得してサンプル作成 |
| `sheet.py <dir> <per>` | 番号付き contact sheet を生成（目視ラベル付け用） |
| `audit_titles.py <types> <out.json>` | タイトルだけ取得してアクセサリ混入を監査 |
| `eval_exclude.py <titles.json>` | 除外語の効果と**誤検出**を出力（出荷前に必ず実行） |
| `test_ng_matrix.py` | 楽天 `NGKeyword` の意味論と語の効きを実機検証 |
| `verify_search.py` | パッチ後の `search()` をカテゴリ横断で検証 |
| `verify_prod.py` | **デプロイ後の本番** `/collect` を検証 |

ローカル実行には `.env` の `RAKUTEN_APP_ID` / `RAKUTEN_ACCESS_KEY` と
`RAKUTEN_REFERER=https://roomstudio.jp/` が必要。楽天APIは約1QPS制限があるので、
連続実行時は `time.sleep(3)` を挟むこと（429 を実際に踏んだ）。

**サンプル画像（96枚）は保存していない**（楽天規約：画像を保存しない）。`fetch_samples.py`
で再取得できる。
